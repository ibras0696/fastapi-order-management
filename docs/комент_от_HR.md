Пример ответа от тим лида, когда разработчик сделал задание не корректно: "при недоступности Kafka или Redis весь сервис падает. Нет graceful degradation. При создании заказа сначала сохраняется в БД, потом отправляется в Kafka. Если Kafka недоступна, запрос падает с 500, хотя заказ уже в базе. Событие не отправлено - консьюмер его не обработает. Получается заказ создан, но не обработан. Нужен retry или хотя бы логирование ошибки. 
 При получении заказа сначала идет запрос в Redis. Если Redis упал, весь API возвращает 500. Нет fallback на БД - один упавший сервис ломает все. В production нужно переключаться на БД при ошибке Redis или обрабатывать исключение. Один и тот же блок сериализации Order в JSON скопирован три раза в разных методах. Если нужно изменить формат, править придется в трех местах. Это нарушение DRY - код должен быть в одном месте. 
 Поле items в заказе это просто список словарей без валидации. Можно отправить пустой список, можно без нужных полей - заказ создастся. Нет проверки, что это реально товары с нужными полями. 
 В consumer и celery задачах используется print() вместо логирования. В проекте есть loguru, но в критических местах print. В production это не попадет в логи, нельзя фильтровать по уровням. Это базовая практика - использовать структурированное логирование. 
 Проект показывает понимание технологий, но не готов к production. Для нашей команды важны не только знания фреймворков, но и понимание production-ready практик.""
